# 🤖 AIエージェント開発指示書 (AI Development Instructions for kanji-pay)

本ドキュメントは、AIコーディングアシスタント（エージェント）が `kanji-pay` プロジェクトで自律的にタスクを遂行する際、厳格に遵守しなければならない行動規範およびプロセスを定義したスキル指示書です。

---

## 🎯 1. イシュー (Issue) の自律起票ルール

AIエージェントが新しい機能開発やリファクタリングのタスクのために GitHub 上に Issue を自動起票する際、以下のルールに従います。

*   **事前の複数起票**: 将来的なマイルストーンや開発ロードマップ（例: 後続の機能追加、関連リファクタ）が明確になっている場合は、目の前のタスクのイシューだけに限らず、**関連する後続のイシューもこのタイミングで一括して事前に起票する**こと。これにより、プロジェクト全体の透明性を高め、マイルストーンの全体像を GitHub 上に可視化します。
*   **要件・設計変更時の自動更新**: ユーザーとの対話によって、実装開始前に重大な設計変更（ディレクトリ構成など）が決まった場合は、実装開始前に必ず **`gh issue edit <Issue番号>`** コマンドを使用して、GitHubイシューのDescriptionを「最新の真実」に自動でアップデートすること。これにより、重い仕様書ドキュメントの二重管理コストを完全に排除しながら、GitHub上に正しい「意思決定の歴史（ログ）」を残します。
*   **必須フォーマット**: イシューを起票する際は、**必ず以下の4つのセクションをすべて含んだフォーマット**で記述しなければなりません。

```markdown
## 🎯 背景 (Background)
*   「なぜこのタスクが必要なのか？」というビジネス的・システム的背景を簡潔に記述する。
*   既存のペインポイントや、将来に向けた目的を記述する。

## 📋 概要 (Overview)
*   「このタスクで具体的に何をするのか？」を箇条書きで分かりやすく記述する。

## 🛠️ 実装方針 (Approach)
*   変更または作成する具体的なファイルパス、ロジック設計、モックの構成方法などを明記する。

## 🧪 テストケース設計 (Test Cases)
*   このタスクの合格基準（Acceptance Criteria）となる、網羅的なテストケースを日本語のBDDスタイルで箇条書きにする。
```

## 📊 2. 開発プロセスの鉄則 (Human-in-the-Loop)

AIエージェントは、コードの変更や削除を行う前に、以下のワークフローを完全に遵守しなければなりません。

1.  **実装計画の提案 (Planning Mode)**:
    *   コードを書き始める前に、必ず `implementation_plan.md` アーティファクトを作成（または大幅アップデート）する。
    *   計画書の `ArtifactMetadata` の `RequestFeedback` を `true` に設定し、ユーザーに計画の承認を求める。
2.  **ユーザー承認の獲得**:
    *   ユーザーからチャットで明確な承認（「これで進めて」「承認します」など）を得るまで、プロダクションコードの書き換え（`write_to_file` や `replace_file_content`）を行ってはならない。
3.  **Issue起票とブランチ切り**:
    *   承認を得た後、上記のフォーマットに従って GitHub Issue を作成（`gh issue create`）する。
    *   ブランチ命名規約（例: `feature/issue-<Issue番号>-<概要>`）に従って、ローカルでブランチを切る（`git checkout -b <ブランチ名>`）。

## 🌿 3. Git・ブランチ命名規約

作業ブランチの命名およびコミットメッセージは、[docs/development_rule.md](file:///Users/kawakitakeisuke/work/development/dev/kanji-pay/docs/development_rule.md) に定義された規約を完全に厳守しなければなりません。

*   **命名テンプレート**:
    *   新規機能: `feature/issue-<Issue番号>-<概要>`
    *   バグ修正: `fix/issue-<Issue番号>-<概要>`
    *   ドキュメント・整理: `docs/issue-<Issue番号>-<概要>` または `refactor/issue-<Issue番号>-<概要>`
*   **コミットメッセージ**:
    *   必ず `feat:`, `fix:`, `docs:`, `test:`, `refactor:` などのプレフィックスから始めること。
*   **プルリクエスト（PR）の作成**:
    *   PRを作成する際は、必ず本文（body）に `Closes #<イシュー番号>`（例: `Closes #7`）のアソシエーションキーワードを記述すること。これにより、PRマージと同時に自動的にGitHub上の関連イシューがクローズされます。

## 🧪 4. テスト駆動開発 (TDD) の徹底

1.  **テストファースト (RED -> GREEN -> REFACTOR)**:
    *   変更を行う際、または新しいビジネスロジック・API通信ラッパーを実装する際は、プロダクションコードを書く前に必ず失敗するテスト（RED）を書き、その後テストを通すための最小限の実装（GREEN）を行い、最後にコードを綺麗にする（REFACTOR）サイクルを回すこと。
2.  **モックの活用**:
    *   API通信のテスト（`src/services/api.test.ts`など）では、実際のサーバーを起動させずに `fetch` などのモックを活用し、インメモリかつミリ秒単位で高速に実行できるテストケースを構築すること。
3.  **マージ前の検証**:
    *   変更完了後、必ず `npm run test` を実行して全テストがグリーンであることを確認した上で、変更をプッシュしてプルリクエストを作成すること。
